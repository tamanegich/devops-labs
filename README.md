### Документація по розробленому веб-застосунку

Застосунок призначений для створення та відслідковування завдань. Користувач може створити завдання вписаши його назву, та помітити його як виконане за допомогою його id.  
Щоб налаштувати середовище для роботи зі застсунком треба

- Встановити `nodejs`, `npm`, `mariadb`, `nginx`, `git`  
  `sudo pacman -S nodejs npm mariadb nginx git`
- Ініціалізувати MariaDB  
  `sudo mariadb-install-db --user=mysql --basedir=/usr --datadir=/var/lib/mysql || sudo systemctl start mariadb`
- Створити БД та користувача
- Клонувати цей репозиторій
- Запустити міграцію БД  
  `node migrate.js --db-host localhost --db-port 3306 --db-user taskuser --db-password taskpassword --db-name taskdb`

#### Щоб запустити застосунок треба застосувати цю команду:

`node server.js --port 3000 --db-host localhost --db-port 3306 --db-user taskuser --db-password taskpassword --db-name taskdb`

#### Документація по API-ендпоінтах

`GET /tasks`

Повертає всі завдання за датою створення, найновіші найперші.

Поля:

- id (integer)
- title (string)
- status (string: pending or done)
- created_at (datetime)

`POST /tasks`

Створює нове завдання і ставить його статус як pending (в процесі).

Request body (JSON): title (string, required).

Повертає статус 201 при вдалому створенні, 400 при відсутності `title` та 500 при помилці сервера.

`POST /tasks/<id>/done`

Позначає завдання як виконане змінюючи його статус на done

Повертає оновлене завдання зі статусом 200 при вдалому оновленні, 404 якщо завдання немає, 500 при помилці сервера.

`GET /health/alive`

Завжди повертає статус 200 з вмістом OK. Підтверджує що сервер працює.

`GET /health/ready`

Повертає статус 200 з вмістом OK якщо підключення до БД стабільне, або 500 з JSON де описується помилка, якщо з'єднання до БД неможливе.

#### (Опціонально) Вхід у ВМ за допомогою SSH

Щоб це було взагалі можливо, то у налаштуваннях ВМ треба зазначити мережевий адаптер як NAT, та в Port forwading додати правило з такими атрибутами:

- Назва: ssh
- Протокол: TCP
- Host IP: 127.0.0.1
- Host Port: <будь-який, головне щоб він був вільний>
- Guest IP: залишайте пустим
- Guest Port: 22

\*Цей момент налаштування може відрізнятись в залежності від софта яким використуєтесь для віртуалізації. При налаштуванні ВМ має бути вимкнена.

Після запуску ВМ вам треба завантажити `openssh` та запустити відповідні сервіси

`sudo pacman -S openssh`  
`sudo systemctl enable sshd`  
`sudo systemctl start sshd`

І потім вже в терміналі основної машини треба ввести таку команду

`ssh -p <Host-Port> <назва-користувача>@127.0.0.1`

І далі ви можете користуватись ВМ зі свого терміналу. Як зручно.

#### (Опціонально)^2 Тестування застосунку з терміналу основної машини

Ні, мова не про те як можна керувати ВМ з терміналу хоста. Можна тестувати напряму з власного терміналу не під'єднуючись нікуди. Для цього треба створити нове правило підключення як вище за такими атрибутами:

- Назва: http
- Протокол: TCP
- Host IP: 127.0.0.1
- Host Port: <будь-який, головне щоб він був вільний>
- Guest IP: залишайте пустим
- Guest Port: 80

І тримати ВМ увімкненою на фоні. Так ви можете використовувати команди нижче, з однією невеликою відмінністю: треба після localhost уточнювати порт, що ви вказали вище, тобто наприклад: http://localhost:8080/tasks

Цим способом неможливо перевірити `GET /health/alive` та `GET /health/ready`, бо nginx блокує всі ендпоїнти не зазначені у бізнес логіці. Щоб його оминути треба уточнювати порт застосунка, що неможливо з хоста.

### Інструкція з тестування розгорнутої системи

Всі перевірки відбуваються з `curl`

Перегляд всіх завдань відбувається цією командою

`curl http://localhost/tasks`

Не рекомендую це запускати найпершою командою для тестування, бо завданнь ще немає.

Щоб створити завдання треба ввести таку команду

`curl -X POST http://localhost/tasks \`  
`  -H "Content-Type: application/json" \`  
`  -d '{"title": "task name"}'`

Після чого вам видасть не дуже гарний рядок чистого HTML коду, який ви можете зберегти в .html файл і побачити, що там є табличка з id, назвою, статусом та часом створення завдання, що ви щойно створили.

Щоб позначити завдання як виконане треба скористатись цією командою:

`curl -X POST http://localhost/tasks/<id>/done`

Де \<id> це id того завдання, яке ви хочете позначити як виконане.

Щоб отримати результат у вигляді JSON, треба зазначити заголовок Accept у команді, бо за замовчуванням `curl` насилає пустий заголовок, а Express буде використовувати HTML, бо він у списку вище. Тобто треба просто після `curl` додати `-H "Accept: application/json"` і результат буде у вигляді JSON.

Щоб перевірити `GET /health/alive` та `GET /health/ready` треба обійти nginx, бо він блокує всі ендпоїнти інакші від основних. Для цього треба вказати порт застосунка, за моїм варіантом 3000, і тоді вже система поверне правильні статуси.
